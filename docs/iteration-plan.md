## 二、迭代规划

### 迭代 0：基础就绪（1-2 天）

**目标**：后端可运行、数据库有数据、管理后台可用

| # | 任务 | 交付物 | 验证标准 |
|---|------|--------|----------|
| 0.1 | 执行 SQL 脚本 | 数据库 8 张表 + 菜单 + 字典 + 示例数据 | MySQL 中可查询到 3 条逝者数据 |
| 0.2 | IDEA 启动后端 | 后端运行在 18080 端口 | Swagger/接口可访问 |
| 0.3 | 接口冒烟测试 | 核心接口可返回数据 | `/api/deceased/list`、`/api/deceased/{id}` 正常返回 |
| 0.4 | 管理后台前端 7 个页面 | ruoyi-ui 下 memorial 模块完整 | 菜单可见、CRUD 操作正常 |
| 0.5 | 管理后台菜单配置 | 后台菜单中可见"纪念馆管理" | 点击菜单进入对应页面 |

**关键文件**：
- `sql/memorial_ddl.sql` → `sql/memorial_menu_dict.sql` → `sql/memorial_sample_data.sql`
- `ruoyi-ui/src/api/memorial/*.js`（7 个 API 文件）
- `ruoyi-ui/src/views/memorial/*/index.vue`（7 个页面文件）

**风险点**：
- 数据库连接配置需确认（application.yml 中 datasource）
- 微信登录需配置 appid/secret，否则 `/api/wxLogin` 不可用
- SMS 验证码 P0 阶段直接返回，需确认 Redis 已启动

### 迭代 1：Phase C 共享组件抽取（2-3 天）

**目标**：抽取 6 个共享组件，提升代码复用性和一致性

| # | 组件 | 功能 | 使用页面 |
|---|------|------|----------|
| 1.1 | NavBar | 自定义导航栏，支持返回/标题/右侧操作 | 所有子页面 |
| 1.2 | MemorialHeader | 逝者头像+姓名+日期+机构信息 | detail/album/video/message/flower |
| 1.3 | AlbumGrid | 九宫格相册展示，支持预览 | album |
| 1.4 | VideoCard | 视频封面+播放按钮卡片 | video |
| 1.5 | MessageItem | 留言条目：头像+姓名+关系+内容+时间 | message |
| 1.6 | FlowerItem | 献花条目：花型图标+姓名+留言 | flower |

**实现要点**：
- 组件放在 `memorial-app/src/components/` 下
- Props 严格类型化（TypeScript），事件用 `defineEmits`
- 现有 5 个页面（album/video/message/flower/detail）重构为使用组件
- NavBar 需处理不同页面的状态栏高度差异

**验证标准**：
- `npm run dev:h5` 编译无报错
- 各页面视觉效果与重构前一致
- 组件可独立复用，Props 类型正确

**参考文档**：`docs/phase-c-components.md`

### 迭代 2：P1 家属功能 — 基础版（5-7 天）

**目标**：家属登录后可创建和管理自己的纪念馆

#### 2.1 家属身份识别（1 天）

| 任务 | 说明 |
|------|------|
| 用户角色区分 | 登录后判断是否为家属（family_user_id 关联逝者） |
| "我的纪念馆"入口 | 个人中心增加"我的纪念馆"卡片 |
| 纪念馆列表页 | 展示当前用户关联的所有逝者纪念馆 |

#### 2.2 创建/编辑纪念馆（2 天）

| 任务 | 说明 |
|------|------|
| 创建纪念馆表单 | 姓名/性别/日期/墓位/简介/封面图 |
| 编辑纪念馆 | 复用创建表单，预填数据 |
| 后端接口对接 | `POST /api/deceased`、`PUT /api/deceased/{id}` |
| 表单校验 | 必填项 + 日期合法性 |

#### 2.3 相册管理（1 天）

| 任务 | 说明 |
|------|------|
| 上传照片 | 图片选择 + 上传到服务器 |
| 删除照片 | 确认后删除 |
| 排序/描述编辑 | 拖拽排序或修改排序号 |

#### 2.4 视频管理（1 天）

| 任务 | 说明 |
|------|------|
| 上传视频 | 视频选择 + 上传 |
| 删除视频 | 确认后删除 |
| 封面/标题编辑 | 编辑视频信息 |

#### 2.5 留言管理 + 二维码（1 天）

| 任务 | 说明 |
|------|------|
| 留言审核 | 待审核列表 + 通过/拒绝操作 |
| 二维码生成 | 前端生成二维码图片，含二维码编码 |
| 二维码分享 | 保存到相册/转发 |

**验证标准**：
- 家属用户登录后可见"我的纪念馆"
- 可完整走通：创建 → 编辑 → 上传相册/视频 → 审核留言 → 生成二维码
- 普通访客不可见管理入口

**参考文档**：`docs/phase-p1-family.md`

## 三、技术债务与风险

### 当前技术债务

| 项目 | 影响 | 建议处理时间 |
|------|------|-------------|
| SMS 验证码 P0 直接返回 | 安全风险，上线前必须接入真实 SMS 服务 | P1 之前 |
| 微信登录 appid/secret 未配置 | 微信登录不可用 | P1 启动前 |
| 无单元测试 | 回归风险 | 迭代 2 过程中逐步补充 |
| 图片/视频上传无压缩 | 存储和带宽浪费 | 迭代 1 |
| 接口无限流 | 高并发风险 | 迭代 3 |

### 关键风险

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 微信支付审核周期长 | 高 | P2 延期 | 提前申请，P2 开始前完成 |
| 图片存储成本 | 中 | 运营成本 | 接入 OSS + CDN |
| 数据库性能（统计查询） | 低 | 响应变慢 | 添加索引 + 统计表定期归档 |

```
迭代 0 (基础就绪)     ── 1-2 天 ──→  ✅ 可运行可管理
```
